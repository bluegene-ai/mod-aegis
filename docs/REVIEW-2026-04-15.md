总体结论：

mod-ac-aegis 的设计方向整体正确，已经具备比较完整的检测面和较强的工程化意识，但当前实现还没有达到“可以放心直接开启自动处罚”的上线标准。优先级最高的问题集中在以下几类：

1. 配置热重载与后台线程之间存在数据竞争。
2. 证据处理热路径存在同步写库，容易直接拖慢 world 主循环。
3. 风险阈值配置未真正接入处罚决策，导致运维调参面和运行时行为不一致。
4. Pull 与 transport 相关检测链路没有完全闭环，存在明确漏报面。

### 代码稳定性 审查报告

**总体评价**：没有看到显式死锁链，常规 Player、Map 空指针防护也比较完整，但当前实现仍有几处会在高负载或热重载下直接伤到线上稳定性的根问题。

**🚨 严重问题**：

- `[src/AcAegisConfig.cpp:57/AcAegisConfig::Reload]`、`[src/AcAegisPersistence.cpp:54/AsyncEventWriter::Enqueue]`、`[src/AcAegisPersistence.cpp:94/AsyncEventWriter::Run]` 配置对象在主线程热重载时被原地改写，后台事件线程同时无锁读取 `eventQueueLimit`、`eventFlushIntervalMs`、`eventBatchSize`。这是标准 C++ 数据竞争，后果是未定义行为，不是“最多读到旧值”那么轻，极端情况下可导致线程异常、错误批次大小甚至 crash。修复建议：把配置改成不可变快照，用原子 `shared_ptr` 交换；或者至少给 `Get/Reload` 加 `shared_mutex`，并把后台线程需要的几个字段复制为独立原子值。
- `[src/AcAegisMgr.cpp:3326/HandleEvidence]`、`[src/AcAegisMgr.cpp:3128/ExecuteAction]`、`[src/AcAegisPersistence.cpp:300/SaveOffense]` 每次证据进入都会走 `ExecuteAction`，而 `ExecuteAction` 无条件调用 `SavePunishState`。即使最终只是 observe、没有累计 offense、没有处罚，也会在主线程做一次 characters 库 upsert，并把大量 `offense_count = 0` 的记录写进 `ac_aegis_offense`。后果是噪声场景下直接拖慢 world 主循环，同时把 offense 表越写越大。修复建议：只在处罚阶段、offense 计数、封禁状态真正变化时落库；普通 observe 证据只保留在内存和异步 event 表。
- `[src/AcAegisMgr.cpp:3785/MaintainOfflinePunishments]`、`[src/AcAegisMgr.cpp:3838/OnWorldUpdate]`、`[src/AcAegisPersistence.cpp:276/LoadExpiredTempBans]`、`[data/sql/db-characters/base/001_ac_aegis_tables.sql:11]` 模块每 5 秒在主线程同步扫一次过期临时封禁，但 offense 表没有 `ban_until` 索引。线上 offense 表一旦变大，这就是固定频率的全表扫描。后果是周期性卡顿和 DB 压力尖峰。修复建议：至少增加 `permanent_ban, ban_until` 联合索引；更好的做法是把离线封禁回收移出 world update，或按“下一条到期时间”调度。
- `[src/AcAegisMgr.cpp:3685/OnLogout]`、`[src/AcAegisMgr.cpp:3807/CleanupOfflineContexts]`、`[src/AcAegisMgr.cpp:1423/DecayRisk]` 离线上下文不会做风险衰减，`CleanupOfflineContexts` 只会删除 `riskScore` 已经降到 0 的上下文；但离线玩家根本不会再调用 `DecayRisk`。结果是任何曾经带风险值下线的玩家上下文都可能长期留在内存里，既是内存泄漏趋势，也会把“应当只在在线期衰减的瞬时风险”跨 relog 带回。修复建议：离线上下文在 `OnWorldUpdate` 中统一做一次 `DecayRisk`，再决定回收；或者直接在 logout 时丢弃瞬时风险，只保留 punish/offense 持久态。
- `[src/AcAegisMgr.cpp:450 / BuildMovementContext]` 传入的 Player* 被直接解引用，缺乏有效的非空校验和生命周期断言。可能导致的后果：在空闲销毁边缘或跨服边界条件时（玩家对象开始析构但在同一 Tick 内仍被抛出移动事件包），将触发空指针引发服务端崩溃。修复建议：在函数入口立即增加 if (!player || !player->IsInWorld()) return; 等激进防御性检查。
- `[src/AcAegisGeometry.cpp:91 / CheckShortSegment]` 对 player->GetMap() 的获取和使用存在盲区。可能导致的后果：玩家在跨地图传送瞬间、副本加载失败、或者掉线临界点时获取到无效的地图指针，导致后续 VMAP 相交查询触发空指针异常。修复建议：显式校验 Map* map = player->GetMap(); if (!map) return false;。
**⚠️ 潜在风险**：

- `[src/AcAegisPersistence.cpp:341/QueueEvent]`、`[src/AcAegisPersistence.cpp:353/DeletePlayerData]`、`[src/AcAegisPersistence.cpp:361/PurgeAllData]` 删除或清空数据时没有和后台事件队列做 barrier，同一批已排队事件可能在 purge 后又重新写回。后果是 GM 看到“已清空”，数据库却回流旧 event。优化建议：在 purge/delete 前先 drain 队列，或者提供按 guid 丢弃 pending event 的能力。
- `[src/AcAegisMgr.cpp:371/IsServerAuthorizedAerialState]`、`[src/AcAegisMgr.cpp:414/HasExternalMobilityAura]`、`[src/AcAegisMgr.cpp:904/BuildMovementContext]`、`[src/AcAegisMgr.cpp:2253/DetectFly]` 每个移动包会重复扫 aura，并在多个 detector 中重复查地面高度。单人开销不大，但百人级高移动密度下会持续占用 world 线程预算。建议缓存已授权空中状态、上次 `groundZ` 与短期 aura 语义，先做 cheap filter，再做地形查询。
- `[src/AcAegisMgr.cpp / OnPlayerMove]` 对玩家内部上下文映射表（_players）的访问缺乏显式互斥锁。优化建议：当前模块极度依赖 AC 的单线程主循环安全假定。随着高并发异步地图特性的推进，此处容易引发数据竞争。建议至少在增删改节点引入读写锁或采用 Lock-Free 结构进行保护。
- `[src/AcAegisGeometry.cpp:54 / RaycastStaticAndDynamic]` 频繁的高精度 VMAP/GObject 线段相交测试在人口密集区或由于外挂高频发包时可能会导致 CPU 帧率暴跌。优化建议：对常规合法的小幅移动引入基于网格或时间窗的碰撞抽样机制，避免每帧高频的全量射线检测。

**✅ 亮点/合理设计**：

- `[src/AcAegisMgr.cpp:897/CanSafelyTeleportForPunish]` 和整个处罚链的 `player/map/session` 判空较完整，没有看到明显的空指针直撞点。
- `[src/AcAegisPersistence.cpp:341/QueueEvent]` 和 `[src/AcAegisMgr.cpp:1668/EnsureFileLogReady]` 到 `[src/AcAegisMgr.cpp:1703/AppendTextLine]` 的日志异步写入方向是对的，说明实现明确在避免同步文件 I/O 直接落到热路径。

### 作弊行为检出能力 审查报告

**总体评价**：文档列出的六大主检测族基本都已落地，外加时间倍率、强制位移和非法坐骑；但仍存在两条关键链路没有闭环，会直接形成漏报或绕过面。

**🚨 严重问题**：

- `[src/AcAegisMgr.cpp:539/HasControlledPullEffect]`、`[src/AcAegisMgr.cpp:555/ControlledMoveTag]`、`[src/AcAegisMgr.cpp:3890/OnSpellCast]` 到 `[src/AcAegisMgr.cpp:3934/OnSpellCast]` 代码里定义了 Pull 受控位移类型，也定义了 `IgnoredPull` 标签，但 `OnSpellCast` 对 pull 类法术只写入 `lastControlledPullMs` 和 spell grace，从未像 Teleport、Charge、Jump 那样调用 `ArmControlledMoveExpectation`。结果是 `DetectControlledMove` 根本不可能产出 `IgnoredPull`，拉拽无效、反击退类外挂会直接漏掉。修复建议：对 `SPELL_EFFECT_PULL_TOWARDS`、`KNOCK_BACK` 等效果建立和 Charge/Jump 同级的期望窗口，按目标点或施法者距离计算最低位移要求。
- `[src/AcAegisMgr.cpp:1138/DetectForceMove]`、`[src/AcAegisMgr.cpp:4094/OnPlayerMove]` 当前 `knockback ACK` 的放行逻辑完全建立在客户端自己报上来的 ACK 包上，没有任何“服务端此前确实下发过击退”的期望绑定。只要外挂构造一个速度字段看起来合理、同时略微移动满足 `expectedFactor`，就能拿到 `recentKnockBackGrace`，从而在 `forceMoveGraceMs` 时间内跳过大部分移动检测。修复建议：把 knockback grace 绑定到服务端已知的受击位移事件，没有 server-issued expectation 的 ACK 不该给 grace，反而应该记作证据。

**⚠️ 潜在风险**：

- `[src/AcAegisMgr.cpp:4058/OnTransportTransition]`、`[src/AcAegisScripts.cpp:277/AcAegisPlayerScript]` 到 `[src/AcAegisScripts.cpp:431/startAcAegisScripts]` 管理器里专门留了 `OnTransportTransition`，但脚本层没有任何 transport/taxi 边界 Hook 去调用它。当前 transport、taxi 的状态同步只能依赖普通 movement 包观察，这对船、电梯、taxi 结束这类边界状态不够稳，会留下漏报和状态串包。优化建议：把 transport attach/detach、taxi cleanup 变成显式边界重置；如果 AC 当前无稳定 hook，就在第一个脱离 transport 的 movement 包上做一次强制 reset。
- `[src/AcAegisGeometry.cpp:75/CheckShortSegment]` 到 `[src/AcAegisGeometry.cpp:115/CheckShortSegment]` 设计文档强调“几何 + reachability”，但当前默认仅在 `allowHotPathReachability` 开启时才做 `CanReachPositionAndGetValidCoords`。默认配置下，单段 noclip 基本还是 `LoS` 阻挡判定为主，对复杂 WMO、门、模型编辑场景的覆盖没有文档写得那么完整。优化建议：至少对重复命中的 blocked segment 做二阶段 reachability 复核。
- `[src/AcAegisMgr.cpp / DetectClimb]` 依赖 climbMinSlopeRatio (约60度)。优化建议：外挂端可以通过多次（每0.1秒微调一点）发送极小段的高Z轴变化包来绕过。建议补充“滑动窗口累加法”，以监控数秒段内的累计垂直位移梯度异常。
- `[src/AcAegisMgr.cpp:510 / 处理 MOVEMENTFLAG_FALLING]` 对处于下落状态的垂直速率校验过于宽松/放行。可能导致的后果：作弊者可以通过持续发送携带 Falling 标志的移动包，绕过地面移速限制进行半空违规位移，或实现“滞空缓降”、“垂直坠界”等未被检测的飞行行为。修复建议：补充对 Z 轴垂直物理速度的校验（重力加速度公式计算），超出正常下落上限的速度需被拦截。

**✅ 亮点/合理设计**：

- `[src/AcAegisMgr.cpp:1954/DetectTeleport]` 同时覆盖 `CoordinateTeleport`、`TeleportBurst`、`StationaryCoordinateShift`，对“小步推进规避大瞬移阈值”的变种有针对性。
- `[src/AcAegisMgr.cpp:2080/DetectNoClip]` 的 `BlockedMicroPath` 思路，对累计微步穿墙而不是单段大跨越的外挂有实战价值。
- `[src/AcAegisMgr.cpp:2238/DetectFly]`、`[src/AcAegisMgr.cpp:2382/DetectWaterWalk]`、`[src/AcAegisMgr.cpp:2488/DetectMount]` 说明模块不只盯着传统 speed/teleport，还覆盖了水面行走、室内强制坐骑、滞空等高频私服作弊变种。

### 检测有效性 审查报告

**总体评价**：宽限窗口、多次命中和合法位移放行思路比传统“一包一判”模块成熟，但当前真正控制处罚落点的闸门存在硬缺陷，所以“低误报”目标还没有被落实到代码层。

**🚨 严重问题**：

- `[src/AcAegisConfig.cpp:206/AcAegisConfig::Reload]` 到 `[src/AcAegisConfig.cpp:211/AcAegisConfig::Reload]`、`[src/AcAegisMgr.cpp:3005/DetermineAction]` 到 `[src/AcAegisMgr.cpp:3125/DetermineAction]` 配置里完整加载了 `Notify`、`Rollback`、`Debuff`、`Jail`、`Kick`、`Ban` 六个风险阈值，但决策函数实际上只用了 `notifyThreshold` 和 `rollbackThreshold`，`debuffThreshold`、`jailThreshold`、`kickThreshold`、`banThreshold` 完全未参与 stage 选择。后果是运维以为自己可以通过调高 `JailThreshold`、`BanThreshold` 来降误报，实际运行时完全无效；像 `CoordinateTeleport`、强 `NoClip`、`LowGravityJump` 这种证据会直接按 cheat type 与 offense tier 升级。修复建议：把风险阈值真正接入 stage promotion，证据类型只能决定最低处罚地板，不能替代风险闸门本身。

**⚠️ 潜在风险**：

- `[src/AcAegisMgr.cpp:660/ResetMovementDetectionState]`、`[src/AcAegisMgr.cpp:3574/OnLogin]`、`[src/AcAegisMgr.cpp:3685/OnLogout]`、`[src/AcAegisMgr.cpp:3977/OnMapChanged]` 生命周期边界没有调用全量检测状态重置，只清了 samples 和 teleport/control 相关窗口，speed、fly、noclip、doublejump 等命中计数仍可能跨 relog、跨 map 延续。后果是玩家在合法 relog 或切图后的第一组可疑包可能被旧 hit window 直接抬成更重证据。优化建议：登录、登出、切图全部调用 `ResetMovementDetectionState`，再按需要补回 teleport/map/jail 相关状态。
- `[src/AcAegisMgr.cpp:2314/DetectFly]`、`[src/AcAegisMgr.cpp:2578/DetectClimb]`、`[src/AcAegisMgr.cpp:4294/HandleDoubleJump]` 到 `[src/AcAegisMgr.cpp:4309/HandleDoubleJump]` 仍有不少关键门槛是硬编码而不是配置项，比如 `flySuspicionHits` 固定要求 3、climb 的 80 到 900ms 时间窗、double jump 的 1.5 码横移和至少 3 次重复命中。尤其 `[src/AcAegisMgr.cpp:4309/HandleDoubleJump]` 直接把 `doubleJumpRepeatHits` 的低于 3 的配置钉死为无效。后果是不同地图、不同延迟、不同移动技能环境下很难精调，配置注释和运行时行为也不一致。优化建议：把这些 magic number 下沉到配置，或者至少把硬下限写进配置说明。
- `[src/AcAegisGeometry.cpp:69/RaycastStaticAndDynamic]` 到 `[src/AcAegisGeometry.cpp:71/RaycastStaticAndDynamic]`、`[src/AcAegisGeometry.cpp:115/CheckShortSegment]` 几何命中点被直接近似成线段中点，调用方却继续把它当作真实 hit 点来计算 `remainingDistance`。后果是 `noClipMinRemainingDistance` 的可调意义被弱化，楼梯、拐角、门口等易抖动位置更难靠配置压掉误报。优化建议：改用真实 hit position API；拿不到真实 hit 点时，就不要再把“剩余距离”做成调参项。
- `[src/AcAegisMgr.cpp:2265 / coordinateJump判定]`  teleportAxisStrongDelta 使用的是硬编码阈值（25.0f）。可能导致的后果：面对幽暗城电梯、高台坠落、大型Boss的击退（Knockback）、被控制强制拉扯位移时，极易产生大范围的瞬间Z/XY坐标跳变引发的高能误报。修复建议：从事件系统中捕获强拉扯或击飞光环（如 SPELL_AURA_MOD_CHARM），结合 MovementInfo::fallTime，给予受控玩家临时的宽容免检窗口期。

**✅ 亮点/合理设计**：

- `[src/AcAegisMgr.cpp:904/BuildMovementContext]` 到 `[src/AcAegisMgr.cpp:1001/BuildMovementContext]` 的 grace 设计比较扎实，覆盖了 spell、teleport、map change、vehicle、transport、fall、root ack、knockback、ghost、authorized aerial 等合法状态。
- `[src/AcAegisMgr.cpp:1021/ComputeAllowedSpeed]` 到 `[src/AcAegisMgr.cpp:1045/ComputeAllowedSpeed]` 在服务端已授权飞行时会回退到 `MOVE_FLIGHT` 基线，不会单纯依赖客户端 `moveFlags` 的 speed type。
- `[src/AcAegisMgr.cpp:4223/HandleDoubleJump]` 到 `[src/AcAegisMgr.cpp:4319/HandleDoubleJump]` 没有做成“单次 jump opcode 即处罚”，而是要求离地、快速重复、正向上升和较低横移同时成立，这个方向是对的。

### 架构与实现合理性 审查报告

**总体评价**：分层思路和 Hook 选点总体合理，模块也确实基本遵守了“不改 AC 核心”的目标；但配置面和运行面的偏移已经开始伤可维护性。

**🚨 严重问题**：

- `[src/AcAegisConfig.cpp:247/AcAegisConfig::Reload]` 到 `[src/AcAegisConfig.cpp:248/AcAegisConfig::Reload]`、`[src/AcAegisMgr.cpp:3109/DetermineAction]` Ban 的两个 AutoAction 配置被加载了，但实际封禁天数使用的是 `Offense.Stage3BanDays` 与 `Offense.Stage4BanDays`。这是典型的死配置/错配置面。后果是运维改了 `Ban.Stage3Days`、`Ban.Stage4Days` 也不会生效，极易形成“配置已调、线上没变”的错觉。修复建议：统一 ban duration 的单一来源；不用的配置项直接删掉，避免继续误导。
- `[src/AcAegisMgr.cpp:4058/OnTransportTransition]`、`[src/AcAegisScripts.cpp:277/AcAegisPlayerScript]` 到 `[src/AcAegisScripts.cpp:431/startAcAegisScripts]` 管理器暴露了 transport 状态边界 API，但脚本层没有任何对应注册。这说明架构设计和实际接线已经发生漂移。后果是后续维护者会误以为 transport 边界已经被显式处理，实际却只能依赖 movement 包副作用。修复建议：要么把 transport/taxi 边界 Hook 真正接上，要么删掉未接线入口，避免出现“假扩展点”。

**⚠️ 潜在风险**：

- `[src/AcAegisGeometry.cpp:59/RaycastStaticAndDynamic]` 到 `[src/AcAegisGeometry.cpp:71/RaycastStaticAndDynamic]` 这个几何抽象一边只返回 `LoS` bool，一边又伪造 `hitX/hitY/hitZ` 给上层消费，接口契约是模糊的。长期看，上层会越来越依赖一个并不真实的几何语义。建议把接口拆成“只判 blocked”与“返回真实 hit 点”两种模式，不要混用。
- `[src/AcAegisMgr.cpp:3128/ExecuteAction]` 到 `[src/AcAegisMgr.cpp:3270/ExecuteAction]`、`[src/AcAegisPersistence.cpp:300/SaveOffense]` 到 `[src/AcAegisPersistence.cpp:341/QueueEvent]` 当前事件写入是异步的，offense 状态写入却是同步的，purge/delete 又没有和异步队列对齐，导致持久化层同时承担了三种不同一致性模型。建议把 punish state 也做成明确的 dirty-write 或状态机提交点，让检测层只负责产出 evidence 和决策，不直接背同步落库负担。

### 建议的修复优先级

建议按以下顺序推进修复：

1. 先修并发与热路径写库问题：配置快照、observe 不落 offense 表、`ban_until` 索引与离线扫表改造。
2. 再修处罚闸门：把 `Debuff/Jail/Kick/Ban` 风险阈值真正接进 `DetermineAction`，同时清理死配置。
3. 最后补检测闭环：Pull 期望窗口、transport 或 taxi 边界 reset、几何真实 hit 点，再做 `Blink`、`Shadowstep`、`Charge`、`Knockback`、`taxi`、`transport`、`waterwalk`、`ghost` 的实机回归矩阵。
